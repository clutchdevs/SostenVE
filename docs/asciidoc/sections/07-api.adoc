[[api]]
== API

=== Contrato vivo (fuente de verdad)

La API es un backend *Hono* montado bajo el base path *`/api/v1`*. El *contrato vivo* se genera en
código a partir de los *esquemas Zod de validación* (vía `zod-to-json-schema`), por lo que los
cuerpos de las peticiones *no pueden desincronizarse* de la validación real:

[cols="1,2",options="header"]
|===
| Recurso | URL
| Especificación OpenAPI 3.1 | `GET /api/v1/openapi.json`
| Swagger UI ("Try it out") | `GET /api/v1/docs`
|===

[TIP]
====
El archivo `docs/02-design/openapi.yaml` es el contrato *de diseño* (fase 02) y usa nomenclatura en
español; puede ir por detrás del runtime. Ante cualquier duda, *la fuente de verdad es
`/api/v1/openapi.json`* (rutas en inglés bajo `/api/v1`).
====

=== Convenciones

* *Base URL:* todas las rutas cuelgan de `/api/v1` (p. ej. `POST /api/v1/auth/login`). En este
  capítulo las rutas se escriben relativas a ese base path.
* *Formato:* JSON en peticiones y respuestas (`Content-Type: application/json`).
* *Idempotencia:* `POST /intake/red-branch` acepta el header opcional `Idempotency-Key` (reintentos
  seguros del formulario offline-first).
* *Localización:* valores de dominio en español (p. ej. `nivel_riesgo: "riesgo_alto"`).

==== Headers

[cols="1,2",options="header"]
|===
| Header | Cuándo
| `Content-Type: application/json` | En todo `POST`/`PUT`/`PATCH` con cuerpo.
| `Authorization: Bearer <JWT>` | Endpoints autenticados (personal). Token de `POST /auth/login`.
| `X-Cron-Secret: <CRON_SECRET>` | Solo `GET/POST /cron/check-sla`.
| `Idempotency-Key: <uuid>` | Opcional, solo `POST /intake/red-branch`.
|===

==== Formato de error

Todos los errores comparten una envoltura uniforme:

[source,json]
----
{ "error": { "code": "VALIDATION_ERROR", "message": "Invalid request", "details": { } } }
----

[cols="1,2",options="header"]
|===
| Código HTTP | Significado
| 400 `VALIDATION_ERROR` | Cuerpo inválido; `details` trae el detalle de Zod.
| 401 | No autenticado / token inválido o secreto de cron inválido.
| 403 | Autenticado pero sin permiso para el recurso (rol o caso ajeno).
| 404 `NOT_FOUND` | Recurso inexistente.
| 409 | Conflicto de estado (p. ej. aceptar/cerrar un caso en estado inválido).
| 422 `DOMAIN_RULE_VIOLATION` | Regla clínica violada (p. ej. TEPT < 4 semanas).
| 429 | Límite de tasa (intake y login).
| 500 `INTERNAL_ERROR` | Error inesperado (nunca filtra detalles).
|===

=== Autenticación

* *`bearerAuth`* (JWT): personal autenticado (psicólogo, coordinador, admin). El token se obtiene en
  `POST /auth/login`. Cada request revalida la *versión del token* contra la cuenta (se invalida al
  reloguear o cambiar contraseña — RF-2.7).
* *`cronSecret`* (`X-Cron-Secret`): solo el endpoint de cron.
* Los endpoints de *intake, líneas de crisis, consentimiento y PAP* son públicos (el solicitante no
  tiene cuenta).

Rutas públicas, *limitadas por tasa* (`intake_requests_per_minute`, 10/min por IP).

==== `POST /intake/triage`
Pregunta Likert inicial; devuelve la rama.

[source,json]
----

{ "respuesta_likert": 1 }

// 200

----

Rama Roja: crea el caso de riesgo alto y devuelve las líneas de crisis. Header opcional
`Idempotency-Key`. `sub_canal: "llamar"` no crea caso.

[source,json]
----
// Request
{
  "sub_canal": "whatsapp-silencioso",   // llamar | recibir-llamada | whatsapp-silencioso
  "nombre": "María",                     // opcional
  "contacto": "+584141234567",           // +58 obligatorio para los canales de contacto
  "edad": 34,                            // obligatorio (0–120)
  "respuesta_likert": 1                  // opcional (1–5)
}

// 201
{
  "caso_id": "c1a2...",
  "rama": "roja",
  "nivel_riesgo": "riesgo_alto",
  "score_urgencia": 12,
  "lineas_crisis": {
    "activa": { "name": "LAPSI", "phone": "+584242907338" },
    "respaldo": [ { "name": "VEN-911", "phone": "911" } ]
  }
}
----

==== `POST /intake/green-branch`
Rama Verde: clasifica por tags y crea el caso.

[source,json]
----
// Request
{
  "nombre": "Pedro",                     // opcional
  "contacto": "+584141234567",           // +58 obligatorio
  "tipo_solicitante": "familiar",        // victima | familiar | voluntario (opcional)
  "estado": "Yaracuy", "ciudad": "San Felipe",  // opcionales (componen la zona)

=== Autenticación y contraseña

==== `POST /auth/login`
Login de personal. Público (rate-limited: bloqueo tras 5 intentos por 15 min).

[source,json]
----
// Request
{ "email": "psico@ppv.test", "contrasena": "••••••••" }

// 200
{ "token": "eyJ…", "refresh_token": "eyJ…", "rol": "psychologist",
  "roles": ["psychologist", "coordinator"] }
// 401 credenciales inválidas
----

==== `POST /auth/change-password`
Cambia la propia contraseña. *Bearer*. Invalida la sesión actual (reautenticar).

[source,json]
----
// Request
{ "contrasena_actual": "••••••••", "contrasena_nueva": "N7#vClaveFuerte" }
// 204  ·  400 (nueva == actual)  ·  401 (actual incorrecta)
----

La `contrasena_nueva` exige ≥ 12 caracteres con minúscula, mayúscula, número y símbolo.

==== `POST /auth/forgot-password`
Solicita recuperación. Respuesta uniforme (sin enumeración de cuentas).

[source,json]
----
// Request
{ "email": "psico@ppv.test" }
// 202 (se envía correo si la cuenta existe)
----

==== `POST /auth/reset-password`
Restablece con token de un solo uso (TTL 60 min).

[source,json]
----
// Request
{ "token": "abc123", "contrasena": "N7#vClaveFuerte" }
// 204  ·  400 (token inválido o expirado)
----

=== Reportes de casos cerrados

Solicitados por la Federación (issue #169, ADR-0017) para revisar y descargar el trabajo ya cerrado. Los tres endpoints comparten filtros, formato y reglas, y solo cambian en cómo entregan el resultado.

*Bearer* (coordinador/admin). Un psicólogo recibe `403`: su acceso sigue limitado a los casos que atendió.

Tres decisiones gobiernan esta sección:

* *Sin identidad del solicitante.* El reporte nunca hace `join` con `case_contacts`; el nombre y el contacto son estructuralmente inalcanzables desde aquí. Lo demás se entrega tal como quedó guardado, incluidos los campos clínicos descifrados.
* *Todo acceso queda auditado.* Cada consulta y cada descarga escriben en la bitácora inmutable el actor, los filtros aplicados y cuántas filas se llevaron (`closed_case_report_view:…` / `closed_case_report_download:…`). Es la trazabilidad que hace aceptable una exportación masiva.
* *Etiquetas legibles, no códigos.* Es un reporte que lee un psicólogo: `finalizado` sale como "Proceso finalizado (objetivos cumplidos)", `ansiedad_estres_agudo` como "Ansiedad / estrés agudo". Un código todavía sin catalogar se presenta humanizado (`algun_motivo_nuevo` → "Algun motivo nuevo") en lugar de filtrar el valor crudo.

[cols="1,2",options="header"]
|===
| Query | Uso
| `desde` / `hasta` | Rango por fecha de cierre (`YYYY-MM-DD` o ISO).
| `nivel_riesgo` | `riesgo_alto` \| `riesgo_moderado` \| `seguimiento`.
| `voluntario_id` | Psicólogo que cerró el caso.
| `motivo_cierre` | Código del motivo de cierre.
| `derivacion_tipo` | `urgente` \| `seguimiento` \| `ninguna`.
| `limit` (1–500) / `offset` | Solo la vista JSON. Las descargas ignoran la paginación y traen todo lo filtrado, con un tope de 5000 filas que la interfaz avisa cuando se alcanza.
|===

Un filtro inválido responde `400` con el detalle de validación.

=== Operación

==== `GET /metrics`
Métricas de SLA (tiempo de asignación por riesgo), cola y uptime. *Bearer* (coordinador/admin).

[source,json]
----
// 200
{ "generated_at": "…", "uptime_seconds": 1234,
  "tiempo_asignacion": {
    "riesgo_alto": { "count": 10, "p50_seconds": 45, "p95_seconds": 300, "avg_seconds": 80 },
    "riesgo_moderado": null, "seguimiento": null } }
----

==== `GET /cron/check-sla` · `POST /cron/check-sla`
Barrido de SLA + asignación (respaldo del escalado event-driven). Header `X-Cron-Secret`.

[source,json]
----
// 200 { "escalated": 1, "assigned": 3 }  ·  401 secreto inválido
----
